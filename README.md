# Hostack 🍯 — ресторанная система

Сайт-ресторан с тремя ролями (гость / пользователь / админ), реальной интеграцией склада и меню, оформлением заказов и управлением статусами.

## Что внутри

```
Hostack-fixed/
├── backend/        — Spring Boot 3.2.5 + MongoDB + JWT
└── frontend/       — React 18 + Vite + react-router-dom
```

## 🔧 Что было исправлено (по сравнению с оригиналом)

### Backend

* **`UserRepository.java` содержал две интерфейса в одном файле** (`UserRepository` + `OrderRepository`) — это нелегально в Java. Оба разнесены по своим файлам.
* **Отсутствовали целые классы**: `User`, `Ingredient`, `OrderItem`, `DishStatus` — упоминались, но не существовали. Переписаны в чистом виде.
* **Дублировался репозиторий** для `MenuItem` (`MenuRepository` + `MenuItemRepository`) — оставлен один.
* **Поля моделей не совпадали с вызовами в сервисах**: `MenuItem` имел `complexity`/`tags`, но сервисы вызывали `getCategory()`/`getOldPrice()`/`getImage()`; `InventoryItem` имел `minThreshold`, но сервисы вызывали `getMinQuantity()`; `Customer` имел `totalSpent`, но сервисы вызывали `getTotalOrders()`; `Discount` имел `percentage`, но сервисы вызывали `getPercent()` и несуществующий `findByTypeAndActiveTrue()`. Все модели переписаны под спецификацию.
* **`Order` не имел геттеров/сеттеров**, нормального конструктора и поля `customerId` — теперь полноценная модель.
* **Сущности `Customer` и `Discount` убраны** — их нет в спецификации; роль клиента выполняет `User`.

### Frontend

* **`Navbar` сравнивал `role === "user"`** (нижний регистр), но backend возвращает `"USER"` — все проверки роли проваливались. Используется верхний регистр + контекст.
* **`MenuPage` / `InventoryPage` / `OrderHistoryPage` использовали хардкод-массивы**, а не API — переписаны на реальные fetch-запросы.
* **`CartPage.placeOrder()` был просто `alert`** — теперь реально вызывает `POST /api/orders`.
* **Отсутствовали страницы**: панель админа, управление блюдами, управление заказами — добавлены.
* **Не было защищённых маршрутов** — добавлены `<ProtectedRoute>` и `<AdminRoute>`.
* **Расхождение ключей в localStorage**: вход сохранял `name`, а `ProfilePage` читал `userName` — единый ключ `hostack_auth` через `AuthContext`.

## 🚀 Запуск

### Требования
- **Java 17+** (для backend)
- **Maven 3.8+**
- **Node.js 18+** + npm (для frontend)
- Подключение к MongoDB Atlas (строка уже зашита в `application.properties`)

### Backend

```bash
cd backend
mvn spring-boot:run
```

Backend стартует на `http://localhost:8080`.

При первом запуске **DatabaseSeeder** автоматически наполнит базу:
- 1 админ-аккаунт (`admin@gmail.com` / `Admin123`)
- 43 продукта на складе
- 33 блюда с рецептами (бургеры, пицца, напитки, салаты, десерты)

### Frontend

```bash
cd frontend
npm install
npm run dev
```

Frontend стартует на `http://localhost:5173` и автоматически открывается в браузере.

| Роль | Что доступно |
|------|---|
| **Гость** | Главная, меню (только просмотр), вход, регистрация |
| **Пользователь** (USER) | Всё для гостя + корзина, оформление заказа, история заказов, профиль |

```
Email:    admin@gmail.com

```

Любая регистрация по умолчанию создаёт `USER`. Аккаунт с email `admin@gmail.com` автоматически становится `ADMIN` (см. `AuthService.java`).

## 🔄 Бизнес-логика

1. **Склад → меню** — блюдо помечено как `available: true`, только если *все* его ингредиенты есть на складе в нужном количестве. Клиент видит «Нет на складе» для недоступных блюд.
2. **Заказ → склад** — при оформлении заказа сервис проверяет наличие, затем списывает ингредиенты. Если хоть чего-то не хватает — заказ отклоняется с понятной ошибкой.
3. **Тип заказа** — клиент выбирает между **самовывозом** и **доставкой курьером**. Для доставки обязателен адрес.
4. **Статусы для самовывоза:** `Принят → Готовится → Готов → Выдан`. До `Готов` всё **автоматически** (шедулер), затем админ нажимает «✅ Клиент забрал заказ».
5. **Статусы для доставки:** `Принят → Готовится → Готов → Курьер выехал → Доставлено`. Все переходы автоматические (раз в 10 секунд) по расчётным временным меткам.
6. **Тайминги доставки** настраиваются в `application.properties`:
   ```
   restaurant.delivery.accept-to-cook-minutes=1
   restaurant.delivery.fallback-cook-minutes=10
   restaurant.delivery.ready-to-courier-minutes=2
   restaurant.delivery.courier-to-delivered-minutes=8
   restaurant.delivery.scheduler-interval-ms=10000
   ```
   Время приготовления берётся из поля `preparationTime` блюд (максимум по позициям заказа). По умолчанию доставка занимает ~21 минуту.
7. **Отображение клиенту:** на странице «Мои заказы» каждый заказ показан с **визуальным таймлайном** (5 узлов для доставки, 3 для самовывоза) и **живым обратным отсчётом** до следующего статуса (тикает каждую секунду).
8. **Опрос сервера:** активные заказы фронтенд опрашивает каждые 5 секунд, завершённые — каждые 30. Без перезагрузки страницы.

## 💳 Виртуальные карты и оплата

Симуляция платёжной системы — реальные деньги не списываются.

**Логика:**
- В профиле пользователь жмёт «+ Добавить карту». Бэкенд генерирует номер (16 цифр), бренд (VISA/MasterCard), срок действия (через 2–5 лет) и **случайный баланс** в диапазоне `restaurant.card.min-balance` … `restaurant.card.max-balance` (по умолчанию 50 000 – 500 000 ₸).
- Карта привязана к `userId`. Сервис гарантирует, что любая операция (списание, удаление) проверяет владение через `requireOwnedCard()`.
- Полный номер карты на клиент **не отдаётся** — DTO `CardResponse` маскирует его до `•••• •••• •••• 1234`.
- В корзине пользователь выбирает: «Картой онлайн» или «При получении». Если карта — выбирает конкретную из списка. Карты с недостаточным балансом подсвечены и недоступны для выбора.
- При оформлении заказа `OrderService` сначала проверяет владение и баланс, **затем** списывает со склада, **затем** списывает с карты. Если баланса не хватает — ошибка возвращается до любых необратимых действий.
- Заказ хранит `paid`, `cardId` и `cardLast4` — для отображения в истории и админке.

**API:**
- `GET /api/cards/user/{userId}` — карты пользователя
- `POST /api/cards` (тело `{userId}`) — создать карту со случайным балансом
- `DELETE /api/cards/{cardId}?userId=...` — удалить (только владелец)

**UI:**
- В корзине — оверлей «Списываем оплату» с прогресс-баром, затем «Оплачено!» с галочкой. Параллельно идёт реальный API-запрос.
- В истории заказов и в админке — бейдж «💳 Оплачено картой •••• 1234».

## 🎁 Программа лояльности

Постоянные клиенты автоматически получают скидку на каждый заказ.

**Тиры (по числу заказов пользователя):**

| Тир | Заказов | Скидка |
|-----|---------|--------|
| 🌱 Новичок | 0–2 | 0% |
| 🥉 Бронза | 3+ | 5% |
| 🥈 Серебро | 8+ | 10% |
| 🥇 Золото | 15+ | 15% |

Пороги и проценты настраиваются в `application.properties`:
```
restaurant.loyalty.bronze-orders=3
restaurant.loyalty.silver-orders=8
restaurant.loyalty.gold-orders=15
restaurant.loyalty.bronze-discount=5
restaurant.loyalty.silver-discount=10
restaurant.loyalty.gold-discount=15
```

**Как работает:**
- `LoyaltyService.getDiscountPercentForUser(userId)` смотрит число заказов пользователя и возвращает процент скидки.
- При создании заказа `OrderService` применяет скидку: `Order.subtotal` (до скидки), `Order.discountPercent`, `Order.discountAmount`, `Order.totalPrice` (после скидки).
- Карта списывает уже сумму со скидкой. Аналогично деление сохраняется в истории заказа.
- В профиле — карточка с прогресс-баром «До «Серебра» ещё 2 заказа → −10%».

**API:** `GET /api/loyalty/{userId}` → `{tier, tierName, discountPercent, ordersCount, totalSpent, nextTier, nextTierName, nextTierDiscount, ordersToNextTier}`.

## 👨‍🍳 Умная синхронизация приготовления

Кухня многоплитная — блюда готовятся параллельно. Если в заказе бургер (5 мин) и напиток (1 мин), то наивно бы готовили оба от старта и напиток ждал бы готовый бургер 4 минуты тёплый. Это плохо.

Hostack делает **обратный расчёт**: фиксирует момент `estimatedReadyAt` (старт + макс. время) и для каждой позиции считает `startsAt = estimatedReadyAt − prepTime`. Так бургер начинает готовиться сразу, а напиток ждёт 4 минуты и готовится только последнюю минуту. Все блюда готовы одновременно.

**Реализация:**
- В `OrderLine` добавлены поля `prepTime` (минуты) и `startsAt` (точное время старта)
- `OrderService` при создании заказа считает `startsAt` для каждой строки от `estimatedReadyAt` назад
- Frontend (`CookingSchedule`) отображает план приготовления с тремя фазами на каждую позицию: «Ожидает» → «Готовится» (с прогресс-баром) → «Готово»

## 🔔 Авто-статусы и подтверждение выдачи

**Шедулер** (`OrderProgressionService`) каждые 10 секунд продвигает все нефинальные заказы:

```
PICKUP:   ACCEPTED ─auto→ COOKING ─auto→ READY ─manual→ PICKED_UP
DELIVERY: ACCEPTED ─auto→ COOKING ─auto→ READY ─auto→ COURIER_DISPATCHED ─auto→ DELIVERED
```

**Самовывоз останавливается на READY** и ждёт, что админ нажмёт «✅ Клиент забрал заказ» — система не может знать, пришёл клиент или нет. После нажатия статус меняется на `PICKED_UP` и клиент видит «✨ Приятного аппетита!».

**В админке** — фильтр «🔔 Готов к выдаче» с пульсирующим бейджем выводит на верх все самовывозные заказы, ждущие выдачи. Карточка такого заказа выделена жёлтой рамкой и имеет большую зелёную кнопку подтверждения.

Админ всегда может вручную переопределить любой статус через панель «Ручное управление» — даже отменить авто-прогрессию (например, отметить «Готов» раньше расчётного времени, если повар работает быстрее). Авто-прогрессия не возвращает статус назад.
9. **Виртуальные карты (симуляция):** в профиле пользователь добавляет карту — backend сам генерирует номер, бренд (VISA/MasterCard), срок действия и **случайный баланс** в диапазоне `restaurant.card.min-balance..max-balance` (по умолчанию 50 000 – 500 000 ₸).
10. **Оплата с карты:** при оформлении заказа можно выбрать «Картой онлайн» и одну из своих карт. Backend проверяет владение, проверяет баланс, **списывает сумму**. Если денег не хватает — заказ отклоняется до того, как тронут склад. Покупка чужой карты невозможна — сервер сравнивает `card.userId` с `order.userId` и кидает «Эта карта вам не принадлежит».
11. **Никаких реальных платежей.** Это симуляция — никуда не уходят запросы, никаких платёжных шлюзов. Все операции — это `balance -= amount` в MongoDB.

## 📡 API

### Публичные
- `POST /api/auth/register` — регистрация
- `POST /api/auth/login` — вход
- `GET /api/menu` — все блюда (с признаком `available`)
- `GET /api/inventory` — все продукты на складе

### Аутентифицированные (нужен Bearer-token)
- `POST /api/orders` — оформить заказ (только USER); поддерживает поля `type`, `deliveryAddress`, `cardId`
- `GET /api/orders/user/{userId}` — заказы пользователя
- `GET /api/orders` — все заказы (только ADMIN)
- `PATCH /api/orders/{id}/status?status=COOKING` — сменить статус (только ADMIN)
- `POST | PUT | DELETE /api/menu/...` — CRUD блюд (только ADMIN)
- `POST | PUT | DELETE /api/inventory/...` — CRUD склада (только ADMIN)
- `GET /api/cards/user/{userId}` — карты пользователя
- `POST /api/cards` (body: `{ "userId": "..." }`) — добавить виртуальную карту (баланс генерируется случайно)
- `DELETE /api/cards/{cardId}?userId=...` — удалить карту (проверяется владелец)

JWT-токен передаётся в заголовке `Authorization: Bearer <token>`. Frontend-клиент (`src/api/client.js`) добавляет его автоматически.

## 🐛 Возможные проблемы

* **`Сервер недоступен`** в UI — backend не запущен или не на порту 8080.
* **Ошибка подключения к MongoDB** — проверьте интернет; строка подключения в `backend/src/main/resources/application.properties`.
* **`401 Unauthorized`** при заказе — токен истёк, разлогиньтесь и войдите снова.
* **Картинка блюда не отображается** — поле `image` в записи MenuItem пустое или URL недоступен.
